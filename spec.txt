1. 项目总览与架构哲学
slg-sentinel 是一套面向 SLG（策略类）手游的营销舆情监控系统。系统自动从多个内容平台（B站、YouTube、TapTap、抖音、快手、小红书）采集视频/评论/评测数据，执行情感分析与用户画像推断，并生成周度分析报告。
1.1 三条铁律
以下三条原则具有最高优先级，贯穿所有设计决策。任何实现方案若违反铁律则自动否决。

铁律 ①：零侵入
所有第三方仓库通过 pip 包引入，不修改源码，随时可 pip upgrade。如需定制行为，仅通过子类化、配置注入或子进程桥接实现。


铁律 ②：双模运行
免登录平台（B站搜索/评论、YouTube、TapTap）通过 GitHub Actions 定时执行；需登录平台（抖音、快手、小红书、B站深度功能）仅在本地/服务器执行。两种模式共享同一套代码，通过 --mode actions|local 开关区分。


铁律 ③：CSV 即数据库
全部数据存储使用 CSV 文件，通过 git push 提交到独立的 data 分支。不引入任何数据库依赖。CSV 编码统一为 UTF-8 with BOM（兼容 Excel 直接打开中文）。

2.1 为什么 YouTube 用三个工具
	•	yt-dlp（主力工具）：ytsearch:关键词 搜索 + extract_info(download=False) 元数据提取 + --write-comments 评论导出。155k ⭐，是最稳定的 YouTube 数据提取工具。
	•	scrapetube（频道浏览补充）：scrapetube.get_channel(channel_id, sort_by='popular') 可以按播放量排序获取频道所有视频。yt-dlp 做此操作需要先获取频道页面再逐个解析，不如 scrapetube 直接。501 ⭐，5年历史，491个项目使用，是成熟的中型项目。
	•	youtube-comment-downloader（专项评论采集）：当需要大量采集某视频评论时，比 yt-dlp 的 --write-comments 更高效（yt-dlp 主要是下载器，评论只是附带功能）。1.2k ⭐。
2.2 关于 tubescrape 的 InnerTube API 思路
tubescrape（4 ⭐，2026年3月新项目）使用 YouTube 的 InnerTube API（POST https://www.youtube.com/youtubei/v1/search），核心只依赖 httpx，无需 API Key。虽然不应作为依赖引入（Stars 过低、项目过新），但其方案值得作为技术储备。

技术储备：InnerTube API 调用模式
如果未来 yt-dlp 的搜索功能被 YouTube 限制，可参考 tubescrape 的 InnerTube 调用模式，在我们自己的代码中实现一个轻量级搜索模块：
•  Endpoint：POST https://www.youtube.com/youtubei/v1/search
•  请求体：需要包含 context.client（clientName: "WEB", clientVersion: "2.xxx"）和 query 字段
•  认证：无需认证，只需要正确的 client context header

class BaseAdapter(ABC):

        ...

    @abstractmethod
    def get_comments(self, video_id: str, **kwargs) -> list:
        """获取评论，返回 Comment 列表"""

全部免登录，GitHub Actions 完全可用。
import yt_dlp
import scrapetube
from youtube_comment_downloader import YoutubeCommentDownloader

class YouTubeAdapter(BaseAdapter):
    def search_videos(self, keyword: str, max_results: int = 20) -> list[VideoSnapshot]:
        """搜索视频——yt-dlp ytsearch"""
        ydl_opts = {
            'quiet': True,
            'extract_flat': False,   # 需要完整info
            'skip_download': True,   # 不下载视频

import requests
import time

class TapTapAdapter(BaseAdapter):
    BASE_URL = 'https://api.taptapdada.com'
    HEADERS = {
        'User-Agent': 'okhttp/3.10.0',
    }
    X_UA = 'V=1&PN=TapTap&VN_CODE=593&LOC=CN&LANG=zh_CN&CH=default'

    def get_reviews(self, app_id: str, sort: str = 'new',
                    max_pages: int = 50) -> list[TapTapReview]:
        """获取游戏评论——免登录"""
        reviews = []
        for page in range(max_pages):
            params = {
                'app_id': app_id,
                'sort': sort,        # 'new' 或 'hot'
                'from': page * 10,
                'X-UA': self.X_UA,
            }
            resp = requests.get(
                f'{self.BASE_URL}/review/v1/by-app',
                params=params, headers=self.HEADERS
            )
            data = resp.json()
            if not data.get('data', {}).get('list'):
                break
            for item in data['data']['list']:
                review = TapTapReview(
                    game_id=app_id,
                    game_name='',          # 需从外部传入或单独查询
                    review_id=str(item['id']),
                    author=item['author']['name'],
                    author_id=str(item['author']['id']),
                    score=item['score'],
                    content=item['contents']['text'],
                    device=item.get('device', ''),
                    spent=item.get('spent', ''),
                    ups=item.get('ups', 0),
                    downs=item.get('downs', 0),
                    publish_time=item['updated_time'],

5.4 MediaCrawlerBridge（仅本地运行）

⚠️ 重要警告
MediaCrawler 在 GitHub Actions 中不可用（需要扫码登录）。此适配器仅在本地/服务器环境使用。CLI 调用时必须指定 --mode local。

关键设计：不修改 MediaCrawler 源码，通过子进程桥接。
import subprocess
from pathlib import Path

class MediaCrawlerBridge:
    """桥接MediaCrawler，仅在本地环境使用"""

    PLATFORM_MAP = {
        'douyin': 'dy',
        'kuaishou': 'ks',
        'xiaohongshu': 'xhs',
    }

    def run(self, platform: str, keywords: list[str]):
        """通过subprocess执行MediaCrawler"""
        # 1. 生成临时runner脚本（注入配置，不修改源文件）
        runner_script = self._generate_runner(platform, keywords)
        tmp_path = Path(f'/tmp/mc_runner_{platform}.py')
        tmp_path.write_text(runner_script)

        # 2. 在MediaCrawler目录下执行
        mc_dir = (Path(__file__).parent.parent.parent
                  / 'third_party' / 'MediaCrawler')
        result = subprocess.run(
            ['python', str(tmp_path)],
            cwd=str(mc_dir),
            capture_output=True, text=True, timeout=3600
        )

        # 3. 读取MediaCrawler输出的CSV/JSON
        # 4. 转换为我们的 VideoSnapshot/Comment 数据模型
        # 5. 删除临时脚本

    def _generate_runner(self, platform, keywords):
        """生成临时运行脚本——通过sys.path和config注入，不修改源文件"""
        return f'''
import sys
sys.path.insert(0, '.')
from config import base_config
base_config.PLATFORM = "{self.PLATFORM_MAP[platform]}"
base_config.KEYWORDS = {keywords}
base_config.SAVE_DATA_OPTION = "csv"
# ... 其他配置覆盖
from main import main
import asyncio
asyncio.run(main())
'''
6. GitHub Actions 工作流设计
每个平台对应一个独立的工作流文件，通过 cron 定时触发并错开执行时间以避免 git push 冲突。
6.1 crawl-bilibili.yml
name: B站数据采集
on:
  schedule:
    - cron: '0 16 * * *'   # UTC 16:00 = 北京时间次日 0:00
  workflow_dispatch:        # 允许手动触发

jobs:
  crawl:
    runs-on: ubuntu-latest
    timeout-minutes: 30
    steps:
      - uses: actions/checkout@v4
        with:
          ref: data            # checkout data 分支
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - name: Cache pip
        uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: pip-bilibili-${{ hashFiles('pyproject.toml') }}
      - name: Install dependencies
        run: pip install bilibili-api-python
      - name: Run crawler
        run: python -m src.cli crawl --platform bilibili --mode actions
        env:
          KEYWORDS_FILE: keywords.yaml

7. GitHub Actions 关键约束与设计决策

序号
约束/决策
说明
1
公开仓库免费无限分钟
推荐使用公开仓库。注意不要把 Cookie 等敏感信息提交到代码中。
2
私有仓库 2000 分钟/月
标准 runner，足够每日 3 个平台各 30 分钟（日均 90 分钟 × 30 天 = 2700 分钟，需留意用量）。
3
单 job 最长运行 6 小时
我们的各采集 job 设置 timeout-minutes: 30，远低于此限制。
4
数据存储在 data 分支
与代码分支（main）分离。main 分支：源代码、配置文件、工作流定义。data 分支：CSV 数据文件、周报。避免代码 PR 触发数据采集冲突。
5
Secrets 管理
DEEPSEEK_API_KEY：AI 分析用（周报生成）。BILI_SESSDATA（可选）：B站深度用户画像时用。不要在 Actions 中存储 MediaCrawler 需要的 Cookie——这些平台需要扫码登录，必须在本地执行。
6
错开各平台的 cron 时间
B站 UTC 16:00 → YouTube UTC 16:30 → TapTap UTC 17:00。避免多个 job 同时 git push 造成冲突。
7
workflow_dispatch
所有工作流都支持手动触发（workflow_dispatch），方便调试和紧急数据采集。

11. 注意事项与工程纪律

序号
规则
详细说明
1
礼貌爬取
所有 Adapter 统一限制请求频率。B站和 TapTap 每请求间隔 ≥ 1 秒。YouTube 工具自行管理频率，无需额外限制。
2
错误处理
网络超时、API 限流、数据格式变更都要有 try-except 和日志记录。使用 Python 标准库 logging 模块，日志格式：[%(asctime)s] %(levelname)s %(name)s: %(message)s。
3
幂等性
同一天重复运行不会产生重复数据。通过「日期 + video_id」去重：写入前检查当天 CSV 是否已存在相同 video_id 的记录。
4
Git 冲突预防
每个平台的 Actions 工作流错开至少 30 分钟。git push 前先 git pull --rebase。
5
CSV 大文件
如果单个 CSV 超过 50MB，按月份分片（{YYYY-MM}_videos.csv）。必要时在 .gitattributes 中配置 Git LFS。
6
安全
keywords.yaml 和 targets.yaml 可以在公开仓库中，不含敏感信息。Cookie / API Key 只通过 GitHub Secrets 或本地环境变量注入。绝不在代码中硬编码任何凭证。


给实施者的提醒
本规格书包含了所有必需的架构决策、数据模型、API 调用示例和工作流配置。实施时应按 Phase 0 → Phase 5 的顺序逐阶段推进，每个 Phase 结束后通过验证命令确认产出物正确，再进入下一阶段。如遇到第三方 API 变更，优先在 Adapter 层适配，不要修改 core 层和数据模型。
